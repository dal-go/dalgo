# Idea: Generics-first, context-first typed convenience layer

**Status:** Specified
**Date:** 2026-06-07
**Owner:** alexander.trakhimenok
**Promotes To:** collection-generated-insert, typed-collection
**Supersedes:** â€”
**Related Ideas:** â€”

## Problem Statement

How might we give dalgo a typed, context-first convenience layer that erases the NewRecordWithData / ExecuteQueryAndReadAllToRecords ceremony â€” returning (T, error) and ([]T, error) directly â€” without sacrificing dalgo's no-reflection, no-codegen, cross-paradigm, and serializable-query-AST differentiators?

## Context

Triggered by the P0 flagship (#1) in backstage/competitors/what-to-borrow.md, synthesized from four competitor deep-dives (upper/db, GORM, Bun, ent). The consistent borrow signal is to adopt the ergonomic shape â€” generics, context-first, convenience terminals â€” while rejecting SQL-coupling and un-portable relational machinery. GORM shipped exactly this (v1.30, 2025): gorm.G[User](db).First(ctx) â€” typed return, context required, no out-pointer. dalgo's record/key model is explicit and portable but verbose. A discussion-time finding reshapes the whole API: Go forbids type parameters on METHODS (a method's only type params are its receiver's), so db.Get[User](ctx,id) and db.Collection("users").Get[User](ctx,id) cannot compile â€” which is exactly why GORM's entry point is the free function G[User](db) and not a method. The type parameter must therefore be introduced by a free function or carried by a generic type. A second finding: Go has no type-level annotations (struct tags exist only on fields), so a type's collection name must come from an interface method (CollectionName() string) or an explicit string, never a type tag. A third: a nested collection's full PATH is inherently runtime parent data, so a type can only bind its LOCAL segment; the parent path resolves through the existing dal.Key chain at call time. A fourth, design-deciding finding: dalgo's DB interface embeds ReadSession only (writes go through a transaction whose handle is a ReadwriteSession). Passing the session per call rather than binding it in the constructor therefore lets read terminals demand a ReadSession and write terminals demand a WriteSession â€” so writing through a non-transactional db becomes a compile error, and the handle itself stays a pure, session-less, reusable value.

## Recommended Direction

A purely-additive generics convenience layer in the dal package over dalgo's existing session interfaces â€” the layer adds no reflection or codegen OF ITS OWN (data binding stays in the adapter, exactly as today, where it already uses json/reflect); serializable AST untouched.

(1) Primitive: free generic functions, e.g. dal.Get[T](ctx, s, key), where the type parameter legally enters via the function.

(2) Flagship ergonomic layer: a SESSION-LESS generic handle â€” the interface dal.Collection[T] â€” constructed two ways with distinct constraints. CollectionOf[T CollectionNamer]() resolves the name from the type's CollectionName() method (compile-time enforced); CollectionAt[T any](name) takes an explicit name for ad-hoc types and overrides. The handle carries only path identity (a composed dal.CollectionRef: name + optional parent) and the phantom type T â€” no db. It is a pure, reusable value: declare `var Users = dal.CollectionOf[User]()` ONCE at package level and pass any session per call.

(3) Session passed per call, which makes read/write safety fall out of dalgo's own type system: read terminals take a ReadSession, write terminals take a WriteSession. Because dalgo's DB is a ReadSession only (writes go through a transaction whose handle is a ReadwriteSession), `Users.Get(ctx, db, id)` compiles while `Users.Set(ctx, db, â€¦)` does NOT â€” a write requires a transaction's WriteSession, enforced by the compiler. No read/write handle split is needed.

(4) Convention interface: CollectionNamer { CollectionName() string } (value receiver) for root collections, supplying the local segment.

(5) Nesting: .In(parentKey) composes the existing dal.CollectionRef parent chain for subcollections, e.g. dal.CollectionOf[City]().In(irelandKey).Get(ctx, db, "dublin").

(6) Terminals, each a thin wrapper over the existing dal session methods. The id argument is `any` and accepts EITHER a plain id value (the common case, matching the core's `Key.ID any`) OR an eager `dal.KeyOption` that sets a concrete id â€” `WithID` (explicit) or `WithFields` (composite). No new id type is invented. Reads: Get(ctx, ReadSession, id) -> (T, error), All(ctx, ReadSession) -> ([]T, error). Point writes: InsertWithID(ctx, WriteSession, id, value) -> (*Key, error) for a KNOWN id; Insert(ctx, WriteSession, value, ...dal.InsertOption) -> (*Key, error) for a GENERATED id; Set(ctx, WriteSession, id, value); Update(ctx, WriteSession, id, ...update.Update) with optional preconditions; Delete(ctx, WriteSession, id). Insert is split deliberately: the two id sources (known vs generated) are never both present, so one signature per intent avoids a `nil` id placeholder and keeps the common `Insert*` vocabulary. GENERATED ids are kept OUT of the id slot: dalgo's eager key-options (`WithRandomStringID`/`WithIDGenerator`) assign the id the instant they are applied â€” with a nil ctx and no collision-retry â€” so they must not run inside the layer. Instead the bare `Insert` passes dalgo's deferred, retry-aware `...InsertOption` family (e.g. `WithRandomStringKey(length, maxAttempts)`) and an INCOMPLETE key down to `session.Insert(ctx, record, opts...)`, returning the assigned key (folding in borrow #7, insert-and-rehydrate). When no option is supplied the LAYER injects a default generator (`WithRandomStringKey(dal.DefaultRandomStringIDLength, N)` for a pinned `N`) â€” this default is layer behavior, not dalgo-intrinsic. CRITICAL DEPENDENCY (chosen approach C): today `InsertOption` is a silent no-op in the shipped adapters (`dalgo2memory` discards the variadic; `dalgo2fs` is not implemented) and no production code wires the existing `InsertWithIdGenerator` retry loop. So generated `Insert` REQUIRES adapters to honor `InsertOption` by running id generation + collision-retry at the storage layer (`NewInsertOptions(opts...).IDGenerator()` + `InsertWithIdGenerator`, or a storage-specific equivalent â€” adapters may implement details specific to their backend). This is a deliberate, coordinated adapter change that also fixes a latent dalgo gap (making `session.Insert(..., gen)` work for ALL callers), NOT pure layer sugar. SAFETY GUARD: until an adapter honors `InsertOption` it silently ignores it (and `dalgo2memory` would store a `<nil>` id today), so the bare `Insert` asserts the returned key has a non-nil id and errors otherwise â€” generated `Insert` fails LOUDLY on a non-honoring adapter instead of writing a bad record. (Adapters run generation BEFORE storage, so the storage id is computed only after the generator fills the incomplete key.) Because only the bare `Insert` accepts `InsertOption` and it has NO id slot, a generator still cannot be passed to any id-taking terminal (InsertWithID/Get/Set/Update/Delete) â€” that rejection is COMPILE-TIME, needing no marker interface and no runtime guard. Keying every point write by id keeps writes safe-by-construction (no WHERE-less mass-mutation). All builds a StructuredQuery over the CollectionRef with a per-row record FACTORY (`NewRecordWithIncompleteKey(name, idKind, new(T))` per row, so results never alias) and inherits QueryExecutor portability â€” a backend may return ErrNotSupported â€” so All is a query-capable-backend terminal, not a portability-peer of point Get. On not-found, Get returns the zero T plus dalgo's not-found error taken from the error the session `Get` CALL returns â€” NOT from `record.Error()` (which is nil for not-found by design) and NOT `record.Exists()` (which can panic if the record was never fetched). Count/Exists/First are mechanical read follow-ons.

(7) Batch is opt-in via a separate interface mirroring dalgo's Inserter/MultiInserter split: ManyInserter[T] { InsertMany(ctx, WriteSession, ...Item[T]) error }, where Item[T]{ ID any; Value T } is a dal-native id+value pair. Using a dal-native pair (not record.DataWithID) means batch needs NO `record` import, so the whole layer â€” including batch â€” is cycle-free and lives entirely in dal.

The existing dal.CollectionRef (a serializable query-AST RecordsetSource used widely â€” dtql serialize/deserialize, dbschema, orm, adapters) is REUSED by composition for path/name/parent, never generified â€” `CollectionRef[T]` is illegal (a generic and non-generic type can't share a name) and would break those call sites and conflate the query-source role with a session-bound handle. The free-function primitive is what the handle delegates to; the handle is what actually kills the NewRecordWithData ceremony. The point CRUD terminals (Get/All/InsertWithID/Set/Update/Delete) are pure additive â€” no existing call site changes. The generated bare `Insert` is the one exception: it depends on adapters honoring `InsertOption` (approach C, above), a coordinated storage-specific adapter change.

## Alternatives Considered

- **Method-style entry points** (`db.Get[User](ctx, id)`, `db.Collection("users").Get[User](ctx, id)`) â€” the most fluent-looking shape, and the user's first instinct. Lost because it is **illegal Go**: methods cannot declare type parameters, so neither form compiles. This is the constraint that produced the free-function + generic-type design (the same wall GORM hit, hence `gorm.G[User](db)` and not `db.G[User]()`).
- **Key-at-call-site only** (`dal.Get[User](ctx, db, dal.NewKeyWithID("users", id))`, no handle, no interface) â€” kept as the *primitive* layer the handle delegates to, but rejected as the flagship: it is the most verbose form and does not erase the boilerplate this Idea exists to remove.
- **Single variadic constructor** (`CollectionOf[T any](name ...string)`) â€” one entry point instead of two. Lost because Go cannot conditionally apply a constraint: a variadic name forces `T any`, discarding all compile-time enforcement of `CollectionName()`. Two constructors with *distinct* constraints (`CollectionOf[T CollectionNamer]` vs `CollectionAt[T any]`) is what buys both safety and override.
- **Binding the session in the constructor** (GORM-style `CollectionOf[User](db).Get(ctx, id)`, terser terminals) â€” considered and initially favored. Lost to the session-less, session-per-call design because dalgo's `DB` is a `ReadSession` only (writes require a transaction's `WriteSession`): passing the session per call lets read terminals demand `ReadSession` and write terminals demand `WriteSession`, so writing through a non-transactional `db` becomes a *compile* error â€” read/write safety with no handle split. A session-less handle is also a pure, reusable value (declare once at package level) instead of a stateful object re-created per transaction.
- **Generifying the existing `CollectionRef` into `CollectionRef[T]`** (operate the type we already have) â€” rejected on two hard grounds: Go forbids a generic and a non-generic type sharing a name, so `CollectionRef[T]` cannot coexist with today's `CollectionRef`; and `CollectionRef` is a serializable query-AST `RecordsetSource` used widely (dtql serialize/deserialize, dbschema, orm, adapters), so adding a type parameter (and CRUD/session behavior) would break those and conflate the query-source role with a session-bound handle. The new `dal.Collection[T]` *composes* a `CollectionRef` for path/name/parent instead.
- **A new `IDSpec`/`GetIDValue() []any` type for the id slot** â€” considered for unifying id specification. Rejected because dalgo already provides the pieces: explicit/composite ids via the eager `dal.KeyOption` (`WithID`/`WithFields`) in the `id any` slot, and generated ids via the deferred, retry-aware `dal.InsertOption`/`IDGenerator` family on the bare `Insert`. A value-getter also can't represent generation (no value exists until insert runs). So no new id type is needed â€” existing, tested seams cover every case.
- **Insert: single `Insert(id, value, ...InsertOption)` vs split, and `Add`** â€” a single Insert forces an ugly `nil` id placeholder for the generated case (`Insert(ctx, ws, nil, u, gen)`); a Firestore-style `Add(value)` for auto-id was considered but the `Add` verb was rejected. Chosen: split by intent under the `Insert*` prefix â€” `InsertWithID(id, value)` for a known id, bare `Insert(value, ...InsertOption)` for a generated id. No `nil`, compile-time generator safety (only bare `Insert` accepts `InsertOption`, and it has no id slot), and one consistent vocabulary.
- **Compile-time typed ids** (`Collection[T, ID comparable]`, `Get(ctx, s, ID)`) â€” fully type-checked ids. Rejected for the MVP: it can't express composite (multi-field) ids in a single `comparable`, it adds a second type parameter to every handle/constructor, and `id any` already matches the core's `Key.ID any`. Wrong-id-type becomes a runtime error rather than a compile error â€” the accepted price for terseness and composite-id support.
- **Housing the layer in a subpackage** (`collection.Of`/`collection.At`) â€” a real option while the design still referenced `record.DataWithID` (which would cycle, since `record` imports `dal`). Made unnecessary once the session-less point-CRUD methods touch only `dal` types and batch uses a `dal`-native `Item[T]`: the whole layer is cycle-free in `dal`, where it is most discoverable. `orm.Collection` (a schema interface in another package) does not clash with `dal.Collection[T]`.
- **Field tag on an embedded marker** (Bun-style `_ dal.BaseRecord ` + backtick `dalgo:"collection:users"`) â€” exploits the field-tag loophole since type tags don't exist. Rejected: reading it requires reflection, which directly violates dalgo's no-reflection differentiator.
- **Constructor naming (`For`/`From`/`Of`/`At`)** â€” naming-only iterations. `For` rejected (reads poorly across terminals). `From[T](s, name)` rejected because it is read-shaped: `From[User](tx, "users").Insert(...)` reads backwards (SQL is `INSERT INTO` / `UPDATE â€¦ SET`, not `FROM`), and the explicit-name constructor must serve writes and nesting too. Chosen: the parallel pair `CollectionOf[T]` (name from the type) and `CollectionAt[T]` (explicit name) â€” they read correctly for every read and write terminal, share the `Collection*` prefix so they sort together, and mirror the returned `Collection[T]`/existing `CollectionRef`/`orm.Collection` vocabulary.
- **Verb-first operation builders** (`dal.Insert[User](db).Items(...)`, `dal.Update[User](db).Set(...).Where(...)`, `dal.Delete[User](db).Where(...)`) â€” operation as the entry verb instead of the collection handle. Rejected as the primary surface: (a) it splits the API into two paradigms (verb-first writes vs noun-first reads) and multiplies entry points; (b) the `.Where()` mutation forms are not sugar over dalgo â€” query-based bulk delete/update do not exist in the session interfaces (point-by-key only) and are not portable across KV backends; (c) they reintroduce the borrow doc's Theme-E `WHERE`-less mass-mutation footgun. Batch-insert ergonomics are kept instead as `Collection[T].InsertMany`; verb-first bulk-`Where` mutation, if ever wanted, belongs in its own later Idea behind an `ErrMissingWhereClause`-style guard.

## MVP Scope

The MVP has two coordinated parts. (a) In the dal package: the session-less interface dal.Collection[T] with both constructors â€” CollectionOf[T CollectionNamer]() and CollectionAt[T any](name) â€” a CollectionNamer interface, the read terminals Get(ctx, ReadSession, id)/All(ctx, ReadSession) returning (T, error)/([]T, error), and the point write terminals InsertWithID(ctx, WriteSession, id, value) -> (*Key, error) (known id), Insert(ctx, WriteSession, value, ...dal.InsertOption) -> (*Key, error) (generated id), and Set/Update/Delete(ctx, WriteSession, â€¦) delegating to the existing dal Inserter/Setter/Updater/Deleter, where the id argument is `any` (plain value or an eager WithID/WithFields KeyOption), plus one level of .In(parentKey) nesting composing the existing CollectionRef parent chain. The handle is session-less; the session is passed per call. (b) In dalgo2memory: honor `...InsertOption` (approach C) by running the id generator + collision-retry at the storage layer, so generated `Insert` has a real backend (this also fixes the latent no-op for raw `session.Insert` callers).

Verified by the cross-adapter end2end suite plus dal unit tests proving: Get/All/InsertWithID/Set/Update/Delete round-trip via CollectionOf reading the name from CollectionName(); CollectionAt overrides the name for the same type; a write terminal requires a WriteSession â€” passing a read-only DB is a compile error (verified by signature and a non-compiling example), while the same write inside RunReadwriteTransaction (WriteSession) commits and rolls back on error; a dalgo2memory-SPECIFIC test (NOT the shared cross-adapter `TestDalgoDB`, which is also driven by a gomock `MockDB` that panics on unknown tx names and would break) exercises generated `Insert` (with WithRandomStringKey and with the zero-option default), asserting a COMPLETE non-empty assigned key is returned and the record round-trips â€” proving approach C; the layer's completeness guard is verified by asserting a non-honoring stub adapter makes generated `Insert` error (not silently store a `<nil>` id); a generator cannot be passed to any id-taking terminal (compile-time, since only bare Insert accepts InsertOption); All over a multi-row collection returns distinct (non-aliased) values and Get on a missing id returns the zero T plus the not-found error (from the Get call's error); a one-level subcollection access via .In(parentKey) resolves the correct path; and all pre-existing dal + dalgo2memory + end2end tests stay green. The ManyInserter[T]/Item[T] batch interface and Count/Exists/First are immediately-after follow-ons (now cycle-free in dal).

## Not Doing (and Why)

- Reflection-based collection inference or any reflection in the hot path â€” violates dalgo's no-reflection differentiator; name comes from CollectionName() or an explicit string
- Type-level struct tags for collection name â€” impossible in Go (tags exist only on fields, not types); the CollectionName() interface is the convention path
- Method-level generic entry points (db.Get[T], db.Collection(name).Get[T]) â€” illegal in Go (methods cannot declare type parameters); the type param enters via a free function or the Collection[T] type
- A single variadic constructor CollectionOf[T any](name...) â€” loses compile-time enforcement of CollectionName(); two constructors with distinct constraints (CollectionOf / CollectionAt) is what buys the safety plus override
- Generifying dal.CollectionRef into CollectionRef[T] â€” illegal in Go (a generic and non-generic type can't share a name) and would break its many AST/dtql/dbschema/orm/adapter call sites; the handle composes CollectionRef instead of replacing it
- Binding the session in the handle â€” the handle stays session-less and the session is passed per call, so read/write safety is enforced by the compiler (ReadSession vs WriteSession) and the handle is a reusable value
- Compile-time typed ids (Collection[T, ID comparable]) â€” can't express composite ids and doubles the type parameters on every handle; the id slot is `any` (plain value | dal.KeyOption), matching the core's Key.ID any
- Query-based bulk mutation (verb-first Delete[T](db).Where(...), Update[T](db).Where(...)) â€” a new capability dalgo's session interfaces do not have (writes are point-by-key only), not portable to all backends, and a WHERE-less mass-mutation footgun; a separate later Idea behind an explicit guard if ever pursued
- Associations / eager-loading / preload / graph traversal â€” join-dependent and structurally un-portable to KV/document backends (per the borrow doc)
- Optional typed-accessor codegen (ent-lite, borrow #10) â€” a separate strategic Idea against the no-codegen positioning, layered on this one later
- The other borrow items â€” condition sugar (#4), pagination (#5), insert-rehydrate (#7), observability (#3), migrations/fixtures (#6), soft delete (#8) â€” separate Ideas; this Idea is the generics foundation and its point CRUD terminals (Get/All/Insert/InsertWithID/Set/Update/Delete) only
- Multi-level nesting in the MVP (3+ collection segments) â€” one level of .In(parentKey) for the MVP; deeper nesting is a follow-on on the same Key chain

## SpecScore Integration

- **New Features this would create:** (1) a `dal` generics convenience layer Feature â€” the session-less `dal.Collection[T]` interface, `CollectionOf`/`CollectionAt` constructors, `CollectionNamer` interface, `.In(parent)` (composing `CollectionRef`), the read terminals `Get`/`All`, and the point write terminals `InsertWithID`/`Insert`/`Set`/`Update`/`Delete`. (2) An "adapters honor `InsertOption` for id generation" Feature (approach C) â€” starting with `dalgo2memory`, fixing dalgo's latent no-op so generated inserts work for ALL `session.Insert` callers; other adapters (fs/sql/firestore/datastore) implement it storage-specifically over time. A follow-on Feature for batch (`ManyInserter[T]`/`Item[T]`) and `Count`/`Exists`/`First`.
- **Existing Features affected:** the point CRUD terminals are additive (no existing call site changes; `CollectionRef` reused unchanged; adapters get those ergonomics for free). The generated `Insert` requires the adapter `InsertOption` change above â€” a coordinated, storage-specific change to each adapter, not free.
- **Dependencies:** the existing `dal` session interfaces (`ReadSession`/`WriteSession`), the `Key`/`CollectionRef` path types, the `update` package, and (for generated `Insert` only) adapter support for `InsertOption`. No new external dependency; `dal` does not gain a `record` import.

## Open Questions

_Resolved during ideation:_ session binding (per-call, session-less handle); read/write safety (no handle split â€” `ReadSession`/`WriteSession` terminal params); package ownership (`dal`, cycle-free); reuse of `CollectionRef` (by composition, not generification); id slot typing (`any` = plain value | eager `WithID`/`WithFields` KeyOption, no new id type); `Insert` split into `InsertWithID` (known id) + bare `Insert` (generated via `...InsertOption`), so generators route through `InsertOption` only and are rejected elsewhere at compile time; all insert terminals return `(*Key, error)` (rehydrate); `Get` not-found returns `(zero T, error)`; generated `Insert` uses approach C â€” adapters honor `InsertOption` (storage-specific), starting with `dalgo2memory`, proven by an end2end test. Remaining:

- **Default generator for a zero-option `Insert`** â€” DECIDED: the layer injects `WithRandomStringKey(dal.DefaultRandomStringIDLength, 5)` (5 collision-retry attempts); revisit the attempt count only if real collision rates warrant.
- **Adapter rollout order for approach C** â€” `dalgo2memory` in the MVP; sequence and per-backend semantics for fs/sql/firestore/datastore (some backends generate ids server-side, so "honor `InsertOption`" may mean "defer to the backend"); until an adapter implements it, generated `Insert` must fail clearly there, not silently no-op.
- **Value vs pointer receiver for `CollectionName()`** â€” pick and document one convention (value receiver keeps `CollectionOf[User]()` working without `*User`).
- **How read filtering layers on later** â€” `All` reads the whole collection; a typed query/`Where`/`First` terminal (borrow #4 condition sugar / #5 pagination) is a separate Idea, but `All` and that future terminal should share one internal `StructuredQuery` builder so the seam survives without a redesign.
- **Scope of the first Feature** â€” do batch (`ManyInserter[T]`/`Item[T]`) and `Count`/`Exists`/`First` ship in the first Feature or the follow-on, now that they are cycle-free in `dal`?
